## 3. Unified Layout Strategy

### 3.1 Layout Unification Approach

**Create a shared layout system (`@higgsfield/layout`) that all apps can adopt**, implementing the storyboard studio pattern:
- Full-page app (`100vw`/`100vh`)
- Full-width header spanning entire app
- Consistent navigation structure
- Framework-agnostic design (works with React, Vue, Vanilla JS)

**Target Apps for Layout Unification:**
- **CineGen** — Adapt existing header to match storyboard style
- **LTX-Desktop** — Adapt existing header to match storyboard style
- **ViMax** — Add full header + sidebar implementation
- **Director** — Add full header + sidebar implementation

**Apps Left Standalone:**
- **Root App** — Already implements the reference pattern
- **chatvideo-yucut** — Service wrapper, no local UI
- **rendiv** — Framework/library, different purpose

### 3.2 Shared Layout System Design

#### 3.2.1 Core Layout Components (`@higgsfield/layout`)

**Framework-Agnostic Layout Package:**
- **AppShell**: Root container with `100vw`/`100vh` flex layout
- **Header**: Full-width sticky header component
- **Sidebar**: Collapsible navigation sidebar
- **ContentArea**: Main content container filling remaining space
- **LayoutProvider**: Context/state management for layout responsiveness

**Framework-Specific Implementations:**
- `@higgsfield/layout-react`: React hooks and components
- `@higgsfield/layout-vue`: Vue composables and components
- `@higgsfield/layout-vanilla`: Vanilla JS factory functions

**Layout Structure:**
```css
.app-shell {
  width: 100vw;
  height: 100vh;
  display: flex;
  flex-direction: column;
}

.app-header {
  width: 100%;
  position: sticky;
  top: 0;
  z-index: 50;
  /* Full-width across entire app */
}

.app-body {
  flex: 1;
  display: flex;
}

.app-sidebar {
  /* Fixed or collapsible width */
}

.app-content {
  flex: 1;
  position: relative;
  display: flex;
  flex-direction: column;
}
```

#### 3.2.2 Unified Header Component

**Header Requirements:**
- **Full-width span**: `w-full` across entire app (not just content area)
- **Navigation**: Logo, main nav items, user actions, mobile menu
- **Framework variants**: React, Vue, Vanilla JS implementations
- **Consistent styling**: Same appearance across all apps

**Header Features:**
- Logo/branding on left
- Navigation menu (dropdowns for sub-items)
- User actions (API key, settings) on right
- Mobile hamburger menu
- Active page highlighting
- Sticky positioning

#### 3.2.3 Styling & Theming Integration

**Shared Design Tokens (`@higgsfield/tokens`):**
- CSS custom properties for colors, typography, spacing
- Unified dark theme (`--bg-app: #050505`, `--color-primary: #d9ff00`)
- Framework-agnostic CSS that works with Tailwind or plain CSS

**Layout-Specific Styling:**
- App shell styles (100vw/100vh, flex layout)
- Header styles (full-width, sticky positioning)
- Responsive breakpoints for mobile/desktop
- Consistent z-index layering

**Theme Variants:**
- Support multiple themes (cinematic, electric, default)
- Runtime theme switching capability
- Consistent application across all layout implementations

#### 3.2.4 Navigation Integration

**Unified Navigation System:**
- **Header Navigation**: Logo, main menu items, dropdowns, mobile menu
- **Sidebar Navigation**: Additional navigation items, app-specific sections
- **Framework-Agnostic Events**: Custom events for navigation state synchronization

**Navigation Features:**
- Active page highlighting across header and sidebar
- Dropdown menus for grouped navigation items
- Mobile-responsive hamburger menu
- Keyboard shortcuts integration
- Route change event handling

**App-Specific Navigation:**
- Each app can define its own navigation structure
- Shared header provides consistent branding and user actions
- Sidebar content customizable per app

#### 3.2.4 State Management and Data Fetching

**Recommendation:** Standardize on React Context + useReducer for domain state, with a shared data-access layer.

**Approach:**
1. **`@higgsfield/state` package** providing:
   - `createContextStore<State, Action>(reducer, initialState, name)` — factory for typed context+reducer stores (pattern used by both CineGen and LTX)
   - `useStore(context)` — typed hook with error boundary for missing providers
   - `withUndoRedo(reducer, options)` — higher-order reducer adding undo/redo (CineGen's pattern, useful for both)

2. **Shared data-access patterns:**
   - `createBackendClient(baseUrl)` — typed fetch wrapper with auth injection, retry, timeout (merge LTX's `backendFetch` with CineGen's IPC patterns)
   - `useBackendQuery(endpoint, options)` — hook for data fetching with loading/error states
   - `useBackendMutation(endpoint)` — hook for mutations with optimistic updates
   - WebSocket hook: `useWebSocket(url, handlers)` — connection management, reconnection, heartbeat (merge patterns from both apps)

3. **Electron IPC abstraction:**
   - `@higgsfield/electron-bridge` package wrapping `window.electronAPI` patterns
   - Typed IPC methods for project management, media processing, export
   - Allows the same React code to work in both Electron and (with mocks) in a browser for testing

#### 3.2.5 Authentication and User Session

**Recommendation:** Abstract authentication behind a shared `@higgsfield/auth` package.

**Current state:**
- CineGen: No visible auth (local-first, SQLite storage)
- LTX-Desktop: API key management in `AppSettingsContext`, backend token injection
- ViMax: Client-side user ID in localStorage, Supabase auth optional

**Approach:**
1. **`AuthProvider` context** supporting multiple auth modes:
   - `local` — no auth, user identity from localStorage (CineGen's current approach)
   - `api-key` — API key stored in settings, injected into backend requests (LTX's approach)
   - `supabase` — Supabase auth with JWT tokens (ViMax's approach)
   - `oauth` — future Google/GitHub login

2. **Session persistence:**
   - Encrypted localStorage for API keys
   - Secure keychain integration via Electron's `safeStorage` API
   - Automatic token refresh for Supabase JWTs

3. **Consistent UX:**
   - Shared `LoginModal` / `ApiKeyModal` component
   - Unified settings page for auth management
   - Consistent "signed out" states across all apps

## 4. Trade-offs and Risks

### 4.1 Risks of Layout Unification

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Framework compatibility** — ensuring layout works across React, Vue, Vanilla JS | Medium | Thorough testing of each framework implementation; clear separation of concerns |
| **Breaking existing functionality** — layout changes may affect app-specific behavior | Medium | Incremental migration with feature flags; maintain old layouts during transition |
| **Mobile responsiveness** — full-width header may need mobile adaptations | Low | Implement responsive design patterns from storyboard reference |
| **Performance impact** — additional layout components may affect rendering | Low | Lightweight implementations; performance monitoring during development |
| **Developer adoption** — learning new layout system across teams | Low | Comprehensive documentation; gradual rollout with training |

### 4.2 Trade-offs

- **Shared vs Custom Layouts:** Unified layout ensures consistency but may limit app-specific customizations. Trade-off is between uniformity and flexibility.

- **Framework-Specific Implementations:** Maintaining parallel implementations (React/Vue/Vanilla) increases maintenance burden but enables broader adoption across the codebase.

- **Reference vs Adaptation:** Using Root App as the reference pattern ensures accuracy but may require compromises for apps with different requirements.

- **Full-Width Header Constraint:** The requirement for headers to span the full app width may conflict with app-specific header designs, requiring adaptation.

### 4.3 Layout Unification Benefits

| Benefit | Impact |
|---------|--------|
| **Consistent User Experience** | Users get familiar navigation patterns across all apps |
| **Unified Branding** | Same header appearance and behavior everywhere |
| **Easier Maintenance** | Single layout system to maintain and update |
| **Faster Onboarding** | Developers can work across apps with familiar structure |
| **Better Mobile Experience** | Consistent responsive behavior across platforms |
